TipTip QA Automation Skills:
A Complete Usage Guide

Take a feature from a Jira ticket all the way to a validated, runnable Playwright/Cucumber test using three Claude Code skills that hand off to one another. Assumes working knowledge of Playwright, Cucumber, and Gherkin.

1Overview & The Three-Skill Pipeline

The three skills form a sequential pipeline. Each skill's output is the next skill's input — they were deliberately designed to hand off to one another.

PRD / Jira ticket / plain-text feature
The raw requirement you start from.
▼
shift-left-manual-test
"What should we test?" → tagged Gherkin manual cases + test-data dependency matrix.
▼
automation-script-generation
"Make it runnable." → POM class + step definitions + .feature, placed in the correct platform dirs, index.js updated.
▼
automation-script-validation
"Is it compliant?" → severity-graded P0/P1/P2 audit report. Read-only — modifies nothing.

Each output feeds the next input. The Gherkin from skill 1 is written with steps that map cleanly onto Page Object methods, so skill 2 can scaffold without guessing. Skill 2 produces files in the exact repo layout that skill 3 knows how to audit.

Tip: Validation is not just a final gate. Run automation-script-validation at any point — on hand-written legacy files, on a teammate's branch during code review, or immediately after generation. It never modifies files, so there is no risk in running it often.

3Skill 1 · shift-left-manual-test

Turns a PRD, Jira ticket, or plain-text feature description into structured Gherkin manual test cases — tagged with TipTip's conventions and accompanied by a test-data dependency matrix. Output is shaped for direct consumption by automation-script-generation. Commands: generate (default), review.

Command: generate (default)

Purpose: Produce a complete set of manual test cases (positive, negative, edge, boundary) from a requirement source. When: at the start of test design — the moment a ticket lands, before any automation exists.

Input typeExample prompt
Jira ticket keyshift-left-manual-test generate PT-12345
Confluence URLshift-left-manual-test generate https://tiptip.atlassian.net/wiki/spaces/QE/pages/123456/Guest-Checkout-PRD
Plain-text descriptionshift-left-manual-test generate → paste: "Guest users can purchase a paid digital content item on TWA without an account, paying via Xendit."

A Jira key is fetched via the Atlassian MCP (getJiraIssue); a Confluence link via getConfluencePage; plain text is parsed directly.

Scenario coverage categories

Every requirement is expanded across all four categories, each with a TipTip priority weighting.

Tag families applied

Type, environment, platform, and domain tags attached to every feature/scenario.

…followed by a test-data dependency matrix (premium content, Xendit sandbox, hardcoded OTP 909090, etc.).

Command: review

Purpose: Audit existing manual test cases and suggest what's missing — uncovered negative paths, missing edge values, mis-applied tags, or steps too vague to map to a Page Object method. Input: a path to a .feature file or pasted cases.

shift-left-manual-test review tests/features/e2e/guest-purchase.feature

Constraints to know upfront

  • No real PII, ever. Uses TipTip standard patterns — "909090" for OTP, "8881588080XX" for phone.
  • 25-scenario cap per feature file. If more is needed, the skill splits by sub-domain.
  • No invented element IDs. Steps reference visible text or logical descriptions — mapping to locators is skill 2's job.
  • Generic, reusable steps — user clicks "<button_text>" button, not page-specific phrasing.
Common Mistakes:
  • Writing implementation-coupled steps yourself (user fills #input-email) — defeats the hand-off to skill 2.
  • Expecting Jira fetching without the Atlassian MCP — paste the ticket text as plain input instead.
  • Ignoring the dependency matrix — it tells you what test data must exist before the generated test can pass.
  • Cramming unrelated behaviors into one feature — if you pass 25 scenarios, let the skill split by sub-domain.

4Skill 2 · automation-script-generation

Takes Gherkin (from skill 1, an existing .feature, or a plain description) and generates three production-ready artifacts following the exact repo structure — a POM class, Cucumber step definitions, and a .feature file — then registers the new page class in the platform's index.js. Commands: generate (default), scaffold-page, scaffold-steps.

Before creating anything, the skill reads the platform's index.js to see which page objects already exist, so it doesn't duplicate one that's already there.

Command: generate (default)

Produces all three artifacts in one pass. Pass the Gherkin from skill 1 directly, or point it at a file: automation-script-generation generate tests/features/e2e/guest-purchase.feature.

Sample POM class — locator priority in action

const { expect, Locator, Page } = require("@playwright/test");

class ProductDetailPage {
  constructor(page) {
    this.page = page;

    // 1. CSS ID — preferred
    this.buttonBuy = page.locator("#i-product-buy-button");

    // 2. XPath — when no ID exists
    this.buttonPayNow = page.locator("//button[normalize-space()='Pay Now']");

    // 3. Tag + text filter — last resort for MUI components
    this.voucherMenuItem = page
      .locator(".MuiMenuItem-root")
      .getByRole("li", { name: "menuitem" })
      .getByText("Apply Voucher");

    // TODO: Verify locator — ID assumed from visible text
    this.labelPaymentError = page.locator("#payment-error-message");
  }

  async clickButtonBuy() {
    await this.buttonBuy.click();
  }

  async verifyPaymentError(expectedText) {
    await expect.soft(this.labelPaymentError).toHaveText(expectedText);
  }
}

module.exports = ProductDetailPage;

How index.js gets updated — and why it matters

After creating a new page class, the skill registers it in the platform's index.js. The export is an array of class constructors — not instances:

const ProductDetailPage = require("./product-detail-page.js");

module.exports = [
  // ... existing pages ...
  ProductDetailPage,
];

The BeforeAll hook in cucumber.conf.js iterates that array, does new ClassConstructor(global.page) for each entry, and assigns the instance to a lowercased global:

SignInPage        →  global.signInPage
ProductDetailPage →  global.productDetailPage
Warning: If the class is a file on disk but missing from index.js, the global is never created and your steps throw at runtime — a silent failure that validation catches as XREF-003.

Locator priority — decision flow

Does the element have a CSS ID (#id)?
→ use page.locator("#id") ✓ preferred
No ID — does it have a data-testid?
→ use [data-testid="…"]
No testid — stable text / MUI component?
→ XPath //…[text()] or .MuiMenuItem-root
Last resort — fragile selector
→ getByText() + // TODO + request an ID

Commands: scaffold-page & scaffold-steps

scaffold-page generates only a POM class (plus its index.js registration). scaffold-steps generates only step definitions for an existing page object.

automation-script-generation scaffold-page twa product-detail
elements: buy button, pay-now button, voucher menu, payment error label

automation-script-generation scaffold-steps tests/pages/twa/product-detail-page.js
When to use which:
  • Starting from a feature/scenario, need everything → generate
  • New page exists in the app, no POM yet → scaffold-page
  • POM exists, but steps aren't wired up yet → scaffold-steps

Constraints to know upfront

  • No page.waitForTimeout() — uses built-in waiting (expect().toBeVisible(), waitForSelector(), waitForURL()).
  • CommonJS only — require / module.exports, never import / export.
  • No hardcoded credentials — uses process.env variables.
  • Never invents locator IDs — unknown IDs get a // TODO: Verify locator comment.
  • expect.soft() for non-critical assertions, matching the repo convention.
Note: When a locator changes, edit only the Page Object. Step definitions and feature files reference behavior, not selectors, so they stay stable.

5Skill 3 · automation-script-validation

Performs static analysis on generated or existing automation files and produces a severity-graded audit report. It is read-only — it modifies nothing and runs no tests. Commands: validate (default), validate-page, validate-steps, validate-feature.

InputExample
A single filevalidate tests/pages/twa/product-detail-page.js
A directoryvalidate tests/pages/twa/
Everythingvalidate all

Scoped variants run one layer only: validate-page (POM + index), validate-steps, validate-feature. Cross-reference (XREF) checks require the full validate because they need all three layers loaded at once.

Rules by severity

32 rules total. P0 blocks merge, P1 is a sprint fix, P2 is backlog quality.

Rules by layer

Where the rules live across the four file types plus cross-reference.

Severity distribution per layer

P0 concentrates in Page Objects, index.js, and cross-reference — the places a runtime failure originates.

SeverityMeaningSLAConcrete example
P0 CriticalWill cause a runtime failureFix before mergePage file missing from index.js (XREF-003) — global never created, steps throw.

A realistic audit report snippet

### tests/pages/twa/product-detail-page.js

#### POM-008: No hardcoded waits — P1

Current (line 14):
  async clickButtonPayNow() {
    await this.page.waitForTimeout(5000);
    await this.buttonPayNow.click();
  }

Expected:
  async clickButtonPayNow() {
    await expect(this.buttonPayNow).toBeVisible();
    await this.buttonPayNow.click();
  }

Rationale: Fixed waits make tests slow and flaky. Playwright's auto-waiting
via expect().toBeVisible() resolves the moment the element is actionable.

Cross-reference validation & the silent failure (XREF-003)

  • XREF-001 a page object with methods but no matching step-definition file.
  • XREF-002 a class declared in index.js but never used by any step.
  • XREF-003 a *-page.js file on disk but not listed in index.js.
Why XREF-003 is P0: The file compiles, the class is correct, the steps look right — but because the BeforeAll hook only instantiates classes in the index.js array, the global (productDetailPage) is never created. The test fails at runtime with an undefined-reference error that points nowhere near the real cause.

Step 2 — Generate the automation scripts

Paste that Gherkin into the next command. The skill detects TWA, reads tests/pages/twa/index.js, sees no ProductDetailPage, and creates:

TypeFile

const { Given, When, Then } = require("@cucumber/cucumber");
const { expect } = require("@playwright/test");

When("user buy the product", async function () {
  await productDetailPage.clickButtonBuy();
});

When("user pay the {string} {string} voucher", async function (item, voucherMode) {
  await productDetailPage.clickButtonPayNow();
});

Then("an error {string} should be displayed", async function (message) {
  await productDetailPage.verifyPaymentError(message);
});

Step 3 — Validate before you commit

automation-script-validation validate tests/pages/twa/

The report comes back with one P1 (POM-008, a stray waitForTimeout(3000)). You fix it in the Page Object — steps and feature stay untouched — then re-run validation to confirm ✅ PASS.

Step 4 — Run the test locally

npm run test:local -- --tags '@commerce and @smoke'

When green, you commit. The whole loop — ticket to validated, passing test — happened without leaving Claude Code except to run npm.

Tip: Don't forget the dependency matrix from Step 1. If the premium content item or the Xendit sandbox isn't provisioned in staging, the test fails for data reasons, not code reasons. Validation can't catch that — only the matrix warns you.
