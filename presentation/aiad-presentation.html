Before I show you what we're building, let me paint the picture of what happens when teams adopt AI tools without coordination. I've seen this firsthand. Each engineer picks their own tool, writes their own prompts, makes their own mistakes — and learns in isolation. Knowledge dies with the session. Prompt quality varies wildly. There's no shared standard. This is the fragmented approach, and it fails at scale.
The Problem

What Happens Without Centralization

  • 10 engineers, 10 different prompt styles — no shared quality baseline
  • Knowledge dies with the session — private prompts are never reused
  • Each engineer trains AI independently — repeating the same mistakes
  • No guardrails — quality depends on individual discipline, not systems
  • Cost is invisible — no way to measure ROI or flag waste
Here's the comparison on concrete dimensions. Fragmented adoption gives you fast individual starts but zero compounding. Centralized adoption requires upfront investment — our 7 guides, the aiad-claude repository — but every improvement compounds across the entire team. This is the same principle behind shared libraries and style guides. We're extending it to how we use AI.

Now — what this means for you concretely. For individual engineers: you are expected to use the shared skills as your default workflow. If /pr-description exists, you don't write your own prompt. If a skill produces wrong output, you report it upstream via MR — you don't fix it locally and move on. No private forks. The improvement loop only works if everyone contributes back.
Expectations

What This Means for Individual Engineers

✅ Do

  • Use shared skills as default workflow for covered tasks
  • Report when a skill produces wrong output — open an MR
  • Write task.md for anything touching 2+ files
  • Review AI output before committing — you own the code
  • Share discoveries in #aiad-discussion
  • Keep MCP tokens current — check expiry proactively

🚫 Don't

  • Create private copies of canonical skills
  • Manually re-prompt when a skill already exists
  • Disable hooks because they're "annoying"
  • Run autonomous sessions on unfamiliar codebases
  • Stockpile uncommitted changes across sessions
  • Let wandering sessions burn tokens — kill after 10 min stall
For engineering leads — you have a heavier responsibility. You own the CLAUDE.md and skills for your domain. Backend lead owns Go files, frontend lead owns Next.js files. You're responsible for hooks being committed to active repos before the team uses Claude Code on them. You run the adoption metrics monthly. You pair with new engineers during their first week. You're accountable for making these guidelines real, not just documented.

This slide is non-negotiable. These are boundaries, not suggestions. Production database mutations via Claude — prohibited. Infrastructure changes without human review — prohibited. Security config generation — prohibited. PII operations outside CLAUDE.md rules — prohibited. Auth logic and payment code — allowed but with mandatory extra review. The hooks catch some of this, but the real boundary is engineering discipline. Read this table. Internalize it.
Non-Negotiable

What NOT to Use Claude Code For

Task Status Requirement
Feature code, tests, PR descriptions ✅ Autonomous Standard MR review
Code review (Go / Next.js) ✅ Autonomous Validate assessment
QA test generation ✅ Autonomous Review generated scripts
QA test validation ✅ Autonomous Review report findings
QA prod tests ⚠️ Interactive Read-only assertions only
Auth / authorization logic ⚠️ Interactive Mandatory second reviewer
Payment / financial logic ⚠️ Interactive Domain expert review, int64 for currency
Infrastructure changes (Terraform, K8s) ⚠️ Interactive Senior review before apply
Production database mutations ❌ Prohibited Deployment pipeline only
Security config (API keys, CORS, JWT) ❌ Prohibited Human authorship required
PII operations outside CLAUDE.md rules ❌ Prohibited Follow data access process

⚠️ Hooks are a safety net, not authorization. Engineering discipline is the real boundary.
